# docs/whitepaper/

## 목적
프로젝트의 공식 백서(Whitepaper)를 보관한다. 프로젝트의 비전·문제 정의·해결 방식·기술 구조를 외부에 공개하는 문서다.

## 구조
- 파일명 형식: `{버전 또는 날짜}-{제목}.md` (예: `v1.0-snowflake-whitepaper.md`)
- 초안·리뷰·확정 버전을 구분해 관리한다

## 최종 기획서 구성 (`v1.0-moving-intelligence.md`)

1. **Executive Summary** — 프로젝트 정의, 핵심 문제, 솔루션 요약
2. **고객/문제/VPC** — B2B 타겟 세그먼트, JTBD, Value Proposition Canvas, **본질적 문제 정의: 관계의 리셋** (2-5)
3. **솔루션 & 기술 아키텍처** — 핵심 기능 4가지, Snowflake Tier 1~3 전략, ASCII 아키텍처 다이어그램
4. **데이터 활용 전략** — 4종 데이터셋 활용방안, Join 전략, Data Analysis Insight 3개
5. **비즈니스 모델** — SaaS/CPA/라이선스 수익 구조, TAM/SAM/SOM
6. **MVP & 로드맵** — 해커톤 데모 범위, 6일 일정, 사업화 4단계
7. **경쟁 포지셔닝** — 한국 경쟁사 비교, 글로벌 Pre-Mover 벤치마크, 포지셔닝 맵

## 브랜딩 전략 구성 (`v1.1-branding-strategy.md`)

1. **브랜딩 전략 개요** — B2B 브랜딩 필요성, 95-5 Rule, Byron Sharp Mental/Physical Availability, Owned vs Rented Prominence, 프레임워크 체계
2. **카테고리 디자인** — "Pre-Mover Intelligence" 카테고리 창조, POV 스토리
3. **브랜드 정체성** — Brand Key Model 9단계, Brand Positioning Canvas 공식 및 적용 예시, Aaker 모델, Keller CBBE B2B 변형
4. **네이밍 전략** — Databricks/Palantir 심층 분석, 한국 듀얼 네이밍 3가지 접근법, 도메인 TLD 확보 전략, 10개+ 후보, 독자 용어(Pre-Move Signal™) 전략
5. **포지셔닝** — PGM Solutions 2025 리브랜딩, Snowflake 4단계 타임라인, Bombora/6sense 상세 분석, 국내 5개사 상세 분석, 11개사 종합 비교표, 두 번째 포지셔닝 맵 (선행성 × 한국 특화도)
6. **B2B 고객 저니** — Gartner 2025 수치(10.2채널, 6.8명, 272일), 8단계 저니 맵, CXO vs 실무자 이중 저니, 내부 챔피언 육성 전략, 온보딩 이메일 시퀀스 (Day 0~30), 세그먼트별 메시지
7. **B2C 사용자 저니** — D-day 타임라인 (D-90~D+30), 골든 타임 정의, 브랜드 경험 설계 3원칙
8. **후킹 포인트** — Pain→Gain 전환 5단계 공식, Tier 1 세그먼트별 5개씩, 엘리베이터 피치 (30초/60초/3분)
9. **터치포인트 설계** — 디지털/오프라인/콘텐츠 채널별 가이드
10. **톤앤매너** — Nielsen Norman Group 톤 프레임워크 (4축), 브랜드 보이스, 언어 가이드, 비주얼 방향성
11. **브랜드 빌딩 로드맵** — 4단계 (카테고리 런치→신뢰 구축→스케일→카테고리 킹)

## 규칙
- 팩트에 근거한 내용만 작성한다 (조사·리서치 규칙 준수)
- 외부 공개 문서이므로 내부 구현 세부사항은 포함하지 않는다
- 변경 시 버전 또는 날짜를 파일명에 반영한다 (v1.0 → v1.2 점프 허용, v1.1 은 branding 전용 트랙)
- **핵심 주장(예: "2~4주 선행") 수정 시 반드시 `.omc/research/` 내 검증 보고서와 `docs/specs/dev_spec.md` 변경 이력과 동시에 동기화한다** (`docs/presentation/00_pitch_story.md` 용어·수치 정합성 포함)
- 기술 용어(Track A/B, Dual-Tier, MULTI_SOURCE, TELECOM_ONLY 등)는 사업 문서 톤을 해치므로 whitepaper 에 삽입하지 않는다. 기술 상세는 `docs/specs/dev_spec.md` 로 분리.
