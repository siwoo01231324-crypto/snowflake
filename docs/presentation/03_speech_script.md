## D-1. 코드 (Data Pipeline) [0:00~2:00]

### 슬라이드 구성 (확정)

**헤드라인**: 코드 — Snowflake Marketplace 통합 MART 구축

**레이아웃**: 좌측 불릿 요약 + SQL 결과 / 우측 코드 캡처 2단

**좌측 패널:**
- Marketplace 구독으로 데이터 바로 ETL
- `Preprocessing.py` : 통신 데이터, 유동인구, 카드 매출, 자산 소득 LEFT JOIN 결합
- `MART_MOVE_ANALYSIS` 테이블 생성
- `TELECOM_ONLY` = 22구 (통신 데이터) 3구
- `MULTI_SOURCE` = (영등포구, 중구, 서초구 / 통합 데이터)
- SQL 결과: 영등포구 → MULTI_SOURCE, 강남구 → TELECOM_ONLY

**우측 패널 (코드 캡처 2단):**
- 상단: Spine 생성 — 25개 구 × 월 범위 cross join (`district_spine`, `month_spine`)
- 하단: LEFT JOIN 5테이블 결합 + `DATA_TIER` 컬럼 생성 (`MULTI_SOURCE_CITIES` 기반 분기)

### 발화 스크립트

안녕하세요, 무빙 인텔리전스입니다.

코드부터 보여드리겠습니다.

Snowflake Marketplace에서 데이터를 구독하면 바로 사용할 수 있습니다.
별도 ETL이 필요 없습니다.

오른쪽 코드를 보시면,
`preprocessing.py`에서 서울 25개 구와 월별 범위를 cross join해서 spine을 만듭니다.
여기에 통신 데이터, 유동인구, 카드매출, 자산소득, 부동산 시세를
LEFT JOIN으로 결합합니다.

결과가 `MART_MOVE_ANALYSIS` 테이블입니다.
왼쪽 아래 쿼리 결과를 보시면,
영등포구는 4종 데이터가 모두 있으니 `MULTI_SOURCE`,
강남구처럼 나머지 22구는 통신 데이터만 있으니 `TELECOM_ONLY`입니다.
이 `DATA_TIER` 구분이 뒤에 나올 모델의 자동 분기 기준이 됩니다.

## D-2. 예측 모델 (Dual-Tier ML) [2:00~4:00]

### 슬라이드 구성 (확정)

**헤드라인**: 예측 모델 — Track A (25구 경량) + Track B (3구 정밀) → 단일 UDF 자동 분기

**레이아웃**: 좌측 불릿 요약 / 우측 코드 캡처 3단

**좌측 패널:**
- Track A (서울 25구 전체 / 경량 모델) — 13개 피처
- Track B (영등포구, 중구, 서초구 3구 정밀 모델) — 19개 피처
- 단일 UDF 자동 분기
  - `PREDICT_MOVE_DEMAND` SQL
  - 내부에서 `DATA_TIER` 확인 후 A, B 자동 분기

**우측 패널 (코드 캡처 3단):**
- 상단: 피처 정의 — `FEATURE_A` (13개, 통신 파생) vs `FEATURE_B` (19개, 멀티소스 추가)
- 중단: XGBoost 학습 — Track A (22구) / Track B (3구) 분리 학습 + MAPE 출력
- 하단: Snowflake Model Registry 등록 — `MOVE_DEMAND_TRACK_A`, `MOVE_DEMAND_TRACK_B`

### 발화 스크립트

예측 모델입니다.

오른쪽 상단 코드를 보시면 피처가 두 종류입니다.
`FEATURE_A`는 통신 데이터에서 파생한 13개 피처,
`FEATURE_B`는 여기에 소득, 매출, 부동산 시세까지 더한 19개 피처입니다.

이 피처 차이가 Dual-Tier의 핵심입니다.
Track A는 서울 25구 전체를 13개 피처로 커버하는 경량 모델이고,
Track B는 영등포·중구·서초 3구를 19개 피처로 정밀 예측하는 모델입니다.

중간 코드가 XGBoost 학습입니다.
Track A, Track B를 각각 분리해서 학습하고 MAPE를 출력합니다.

하단을 보시면 학습된 모델을 Snowflake Model Registry에 등록합니다.
`PREDICT_MOVE_DEMAND`라는 단일 UDF를 호출하면
내부에서 `DATA_TIER`를 확인하고 Track A와 B로 자동 분기합니다.
하나의 SQL 함수로 서울 25구 전체를 커버합니다.

> `00_pitch_story.md §4.1`의 lag k=-3~+3 r값 테이블을 PPT에 삽입
> k=+1 행(r=0.895) 강조 표시
> **포인트**: "서울 25구 38개월, n=841, p<0.001"

### 발화 스크립트

솔루션을 정리하겠습니다.

이사는 단순한 주소 변경이 아닙니다.
기존 브랜드와 고객의 관계가 한꺼번에 초기화되는 **리셋 버튼**입니다.
이사자 10명 중 9명이 새 브랜드를 시도할 의사가 있고,
이사 후 3개월 소비는 일반인 3년 소비와 같습니다. 13배입니다.
이사 시점에 타겟팅하면 마케팅 전환율이 일반 대비 2배입니다.

그런데 이 골든 윈도우가 열려 있는 75일에서 90일 동안,
코웨이도, 오늘의집도, KT도 — 이사를 미리 모릅니다.
이사 데이터 응답률은 1주 지나면 30퍼센트 하락하고,
한 달 지나면 절반 이하로 떨어집니다.
전입신고 데이터는 이사 후 최소 2주에서 45일 뒤에야 잡힙니다.
이미 가치가 절반 이상 증발한 뒤입니다.

저희는 이 문제를 해결했습니다.
통신 신규 계약이 실제 개통보다 1개월 먼저 발생한다는 구조적 사실을 발견했습니다.
서울 25구, 38개월 패널에서 상관계수 0.895입니다. p값 0.001 미만.
매달 계약의 28에서 30퍼센트가 당월 미개통 상태로 대기합니다.
이 carryover 구간이 데이터 신선도가 가장 높은, 바로 그 골든 윈도우입니다.

KOSIS 공식 통계와의 교차검증에서는 상관이 거의 0입니다.
이건 실패가 아닙니다. KOSIS가 이사 후 30~45일에 집계되기 때문에 당연한 결과이고,
공공 통계로는 구조적으로 포착할 수 없는 선행성이 우리 데이터의 가치입니다.
한국에 이사 2~4주 전 예측 데이터를 B2B로 제공하는 서비스는 현재 존재하지 않습니다.

**캡처 2 — 기술 확장 아키텍처 (다이어그램)**
> Dynamic Table + Snowflake Task + Cortex Agents 3가지를 도식화

| 기술 | 현재 (수동) | 향후 (자동) |
|---|---|---|
| 마트 갱신 | Snowpark 수동 실행 | **Dynamic Table** 변경 감지 자동 |
| 모델 재예측 | 수동 트리거 | **Snowflake Task** 월간 cron |
| 멀티 AI | Cortex AI 개별 호출 | **Cortex Agents** RAG+Analyst+UDF 오케스트레이션 |

> **포인트**: "자동화 3종 세트 + Marketplace 구독만 늘리면 전국 즉시 확장"

### 발화 스크립트

마지막으로 향후 전략입니다.

저희는 3구 샘플로 개념을 증명했습니다.
같은 Dual-Tier 프레임워크를 25구, 광역시, 전국으로 모델 재설계 없이 확장할 수 있습니다.
Marketplace 구독 하나 늘리면 Track B가 그 구를 즉시 커버합니다.

자동화도 준비되어 있습니다.
Dynamic Table로 마트를 변경 감지 자동 갱신하고,
Snowflake Task로 월간 예측을 스케줄합니다.
Cortex Agents가 출시되면 RAG, Analyst, 커스텀 UDF를
하나의 에이전트 안에서 오케스트레이션할 수 있습니다.

비즈니스 모델은 B2B SaaS 구독입니다.
월 99만 원에서 1,000만 원 구간, 건당 CPA 3만에서 50만 원.
보수적으로 Year 1 ARR 13억 원을 목표로 합니다.

핵심을 하나만 기억해 주십시오.
통신 신규 계약은 실제 이사보다 1개월 먼저 발생합니다.
상관계수 0.895, p값 0.001 미만.
이 1개월이 공공 통계로는 포착할 수 없는 골든 윈도우입니다.

감사합니다.
