### Task Flow (5 Steps)

#### Step 0: 트랙 선택 확정

- 테크 트랙을 기본값으로 확정
- 비즈니스 트랙인 경우: Step 3의 평가 기준을 비즈니스 트랙 기준으로 교체
- **검증**: 계획 문서에 "테크 트랙 기준" 명시 완료

#### Step 1: 리서치 기반 확보

- `docs/background/` 00~16번 문서 17개 전체 숙지
- 섹션별 연결: 경쟁 분석(13, 14) / 시장 규모(15) / 수익 모델(01, 16) / Snowflake 기능(06) / 데이터 활용(07) / 심사 전략(04, 05)
- **검증**: 각 섹션과 연결되는 리서치 문서 매핑 완료

#### Step 2: 프로젝트 기획서 본문 작성 (7개 섹션)

- `docs/whitepaper/v1.0-moving-intelligence.md` 생성
- 7개 섹션 구성:

**섹션 1. Executive Summary**
- 프로젝트 한 줄 정의, 핵심 문제/솔루션/가치 (각 1-2문장)
- 슬라이드 1장 분량

**섹션 2. 고객 / 문제 정의 / 가치 제안 (VPC)**
- 타겟 고객: Primary(이사 예정 개인), Secondary(부동산 투자자), Future(지자체/지역사업자)
- 고객별 JTBD (Jobs-To-Be-Done) 분석
- 현재 Pain Points & 대안 솔루션의 한계
- Value Proposition Canvas: Customer Profile (Jobs/Pains/Gains) + Value Map (Products/Pain Relievers/Gain Creators)
- 슬라이드 2장 분량

**섹션 3. 솔루션 & 기술 아키텍처**
- 핵심 기능: 이사 적기 분석, 지역 생활 점수, 이사 후 투자 인사이트, AI 추천 엔진
- Snowflake 활용 설계: Snowpark(데이터 파이프라인 & ML), Cortex AI(LLM 인사이트), Streamlit in Snowflake(프론트엔드), Marketplace(데이터셋 연동)
- 시스템 구성도 (텍스트 기반)
- 슬라이드 2장 분량

**섹션 4. 데이터 활용 전략**
- 4개 데이터셋별 활용 방안:
  - RICHGO: 아파트 시세 + 인구 이동 + 전입전출 (서울시 전체 100세대 이상 아파트)
  - 아정당: 이사/이동 추정 + 렌탈 가전 수요 + 고객 세그먼트
  - SPH: 유동인구 패턴 + 소비 트렌드 + 자산/소득 수준 (**지리적 제약: 서초구, 영등포구, 중구만 커버 — 3개 구 집중 분석으로 scope 한정**)
  - NextTrade: 지역별 부동산-주식 상관관계 분석
- 데이터 결합(Join) 전략
- 데이터 품질/한계 인식 (SPH 지역 제한 명시)
- 슬라이드 2장 분량

**섹션 5. 비즈니스 모델 & 시장 기회**
- 수익 모델 (Freemium / B2B / 데이터 API 등)
- TAM/SAM/SOM 추정 (공개 데이터 기반, 출처 필수 명시)
- 경쟁 우위
- 슬라이드 1-2장 분량

**섹션 6. MVP & 로드맵**
- 해커톤 데모 범위 (In-scope / Out-of-scope)
- 개발 일정 (04-06 ~ 04-12)
- 향후 확장 계획
- 슬라이드 1장 분량

**섹션 7. 경쟁 포지셔닝**
- 기존 이사/부동산 서비스 비교 (직방, 호갱노노, 네이버 부동산 등)
- 차별화 포인트: 4개 데이터셋 결합 + Snowflake 네이티브
- 슬라이드 1장 분량

- **검증**: 7개 섹션 모두 작성, 4개 데이터셋 각각 최소 1개 섹션에서 구체적 언급, 각 섹션이 1-2 슬라이드 분량, 출처 없는 통계 없음

#### Step 3: 심사 기준 대응 검증 (테크 트랙)

- 테크 트랙 평가 기준 크로스체크:
  - 기술 혁신성 → 섹션 3
  - 데이터 활용 능력 → 섹션 4
  - 문제 해결 능력 → 섹션 2
  - 비즈니스 임팩트 → 섹션 5
  - Snowflake 활용도 → 섹션 3
- 비즈니스 트랙으로 변경 시: 위 매핑을 비즈니스 트랙 기준으로 교체
- **검증**: 5개 평가 기준 모두에 대응 내용 존재

#### Step 4: `.ai.md` 최신화 및 마무리

- `docs/whitepaper/.ai.md` 업데이트 (생성된 파일 반영)
- **검증**: `.ai.md`가 실제 파일 구조를 정확히 반영

### Guardrails

**Must Have**:
- 4개 데이터셋 모두 활용 방안 명시
- Snowflake 네이티브 기능(Snowpark, Cortex AI, Streamlit) 활용 설계
- 타겟 고객 최소 2개 세그먼트 정의 (Primary: 이사 예정 개인, Secondary: 부동산 투자자)
- MVP 범위가 해커톤 마감(04-12) 내 실현 가능한 수준
- SPH 데이터 지리적 제약(서초구/영등포구/중구) 섹션 4에 명시
- 시장 통계에는 반드시 출처 명시
- **[템플릿 필수]** Problem Statement + Snowflake Architecture Diagram + Tech Stack 포함
- **[템플릿 필수]** Data Analysis Insight 3개 (각 최대 3 bullets, 활용 데이터·분석 방법·주요 효과 포함)

**Must NOT Have**:
- 구현 불가능한 과대 비전 (6일 내 불가능한 기능)
- 데이터셋과 무관한 기능 설계
- 내부 구현 세부사항
- 출처 없는 시장 규모/통계 수치 (허위 통계 금지)
